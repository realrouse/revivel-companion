# Changelog — ReviveL Companion v0.2.0

**Date:** 2026-07-16  
**Previous version:** 0.1.5-hotfix1  
**Version files:** `package.json`, `src-tauri/Cargo.toml`, `src-tauri/tauri.conf.json`

---

## Summary

Version **0.2.0** introduces **file staging over Native Messaging**, so the ReviveL browser extension can upload publish payloads through the Companion instead of relying only on browser-side workarounds. It also packages the security and UX work landed across the 0.1.3–0.1.5-hotfix1 line into a clearer minor release.

---

## Added

### Native Messaging — file staging (new in 0.2.0)

The Companion native host now accepts three new message types:

| Type | Purpose |
|------|---------|
| `begin_stage_file` | Create a staging file under the system temp dir (`…/revivel_uploads/`). Accepts `filename`. Returns `{ success, path }`. |
| `append_file_chunk` | Append a base64-encoded `chunk` to an existing staged `path`. |
| `finish_stage_file` | Finalize staging for a `path` and return it for use by the extension / `lbrynet` publish flow. |

Details:

- Filenames are sanitized (alphanumeric, `.`, `_`, `-` only).
- Staging path pattern: `upload_<millis>_<safe_name>` in `std::env::temp_dir()/revivel_uploads`.
- Uses the new `base64` crate dependency for chunk decoding.
- Extra `eprintln!` diagnostics for native-host message types (helps debug extension ↔ host issues).

### Already present (carried from 0.1.x into this release line)

These were introduced before 0.2.0 and remain part of the product:

- **Secure RPC proxy** (`type: "rpc_call"`) via Native Messaging with Basic Auth + Origin (since 0.1.3)
- **`get-rpc-credentials`**, **`open-lbry-uri`**, **`ping`** native messages
- Strict **`allowed_origin`** for the configured extension ID (no `"*"`)
- Random **rpcuser / rpcpass** generated and persisted
- First-time **Extension ID setup modal**, Save / Reset ID in Settings
- **Statistics** tab (download/upload totals and speeds)
- Multi-platform packaging (Linux / Windows / macOS CI and helper scripts)

## Security / integration notes

- File staging is only available to extensions listed in the Native Messaging host `allowed_origins` (known stable ID + user-configured ID).
- Staged files live in the OS temp directory; callers should treat paths as sensitive and clean up when done (Companion does not auto-delete on `finish_stage_file`).
- Direct browser HTTP to `127.0.0.1:5279` still uses extension Origin + credentials when the daemon is started by Companion; privileged flows should prefer NM `rpc_call` / credentials APIs.

## Upgrade notes

1. Install or rebuild Companion **0.2.0**.
2. Ensure the ReviveL extension version you use supports the new staging message types if you rely on publish-from-browser.
3. If using an unpacked/dev extension, re-enter the Extension ID in Settings if needed and click **Save Extension ID** (daemon may restart).
4. Re-run **Register lbry:// + Native Messaging host** if the host path or allowed origins need refresh after install.

## Known limitations

- Staging has no size quota or virus scanning; large uploads depend on disk space in temp.
- Chunks are decoded with best-effort base64 (`unwrap_or_default` empty on bad input) — extension should validate success.
- Cross-platform installers still best produced via GitHub Actions for Windows/macOS signing workflows.
